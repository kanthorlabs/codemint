# CodeMint PRD & Engineering Spec: EPIC-03 (ACP Execution Layer)

## 1. Overview
EPIC-03 defines how CodeMint executes the tasks generated in the Brainstorming Workflow. It acts as the bridge between the Go orchestration backend and the underlying AI CLI agents (e.g., OpenCode, Codex) via standard standard input/output (JSON-RPC over standard I/O).

---

## 2. Core Architecture: The Persistent ACP Worker

### 2.1 Lifecycle Management
- **Single Process:** To prevent high token costs and cold-start latency, CodeMint spawns **one** background `os/exec` process (the Persistent Worker) per active CodeMint Session.
- **Context Clearing:** The Go backend sends a `clear` or `reset` command via the JSON-RPC stream at the boundary of every User Story to flush the agent's context window, keeping token usage lean without killing the process.
- **Teardown:** The `os/exec` process is gracefully terminated only when the user archives or pauses the CodeMint Session.

---

## 3. Security & Autonomy: The Auto-Approval Interceptor

To prevent the "Agent Babysitter" problem while maintaining security, CodeMint intercepts all CLI execution requests from the Agent.

### 3.1 Schema Update: `project_permission`
A new table manages the whitelist/denylist for agent operations.

| Column | Type | Description / Logic |
| :--- | :--- | :--- |
| `id` | TEXT (UUIDv7) | Primary Key |
| `project_id` | TEXT (UUIDv7) | FK to `project.id` |
| `allowed_commands` | TEXT (JSON) | e.g., `["go test", "npm run lint"]` |
| `allowed_directories` | TEXT (JSON) | e.g., `["./src", "./tests"]` |
| `blocked_commands` | TEXT (JSON) | Explicit denylist (e.g., `["rm", "curl"]`) |

### 3.2 Interceptor Logic
When the Agent emits a tool call (e.g., `{"type": "tool-call", "tool": "terminal", "command": "go test"}`):
1. **The Intercept:** CodeMint halts the event from reaching the UI.
2. **Evaluation:** It checks the command against the `project_permission` table.
3. **Match (Whitelist):** CodeMint automatically executes the command locally, captures the output, and sends it back to the Agent via `stdin`. The task status remains `processing`. The user is not disturbed.
4. **No Match / Blocked:** CodeMint halts the agent stream, updates the task status to `awaiting`, and pings the UI for manual human approval.

---

## 4. State Mapping & Event Routing

### 4.1 JSON-RPC to DB Status Translation
CodeMint translates standard ACP JSON-RPC events into our immutable database states:

* **`session.start` / `turn-start`:** CodeMint Task -> `processing`.
* **`tool-call` (Not Whitelisted):** CodeMint Task -> `awaiting` (Requires human input).
* **`human-input-request`:** CodeMint Task -> `awaiting`.
* **`turn-end` / `task-complete`:** CodeMint Task -> `success` (Agent finished, scheduler pulls next task).

### 4.2 Noise Reduction & Event Streaming
Agent execution produces high-frequency "noise" (e.g., `thinking`, `file-edit` micro-events).

* **TUI (Terminal UI):** Subscribes to the raw, high-bandwidth event stream to show real-time "thinking" logs in a dedicated pane.
* **CUI (Chat UI):** Subscribes *only* to terminal/blocking state changes (`awaiting`, `success`, `failure`) to prevent chat spam.
* **The `/summary` Command:** * **Usage:** `/summary [task_id]`
    * **Logic:** CodeMint stores the last N raw events in a circular memory buffer. When called, it aggregates these events into a clean `<thinking>` block.
    * **Fallback:** If no `task_id` is provided, the Go backend queries the database for the active session's most recent `processing` or `awaiting` task and summarizes that context.

## 5. Runtime Wiring (Cross-Epic Integration)

Stories 3.1–3.11 deliver the components in isolation. Stories 3.12–3.18 assemble them at runtime so EPIC-02 (Brainstorming Workflow) and EPIC-04 (Multi-Interface Support) can actually drive the ACP layer.

### 5.1 EPIC-02 Wiring (Stories 3.12–3.16)
* **3.12 ACP Runtime Pipeline Assembly** — wire `Pipeline`, `Interceptor`, `StatusMapper`, `Fanout`, `BufferRegistry`, and `PipelineConsumer` into a single goroutine started at session boot. Today these structs exist but are never connected; the `/acp` REPL command reads `worker.Out()` directly.
* **3.13 Pending Task Scheduler Loop** — pull pending tasks generated by Phase 3 in strict `(seq_epic, seq_story, seq_task)` order and dispatch them; respect the `awaiting` pause; honor story-boundary context resets from Story 3.2.
* **3.14 Executor Task-Type Routing** — branch on `task.type`: Coding → ACP, Verification → `LocalRunner`, Confirmation → human pause, Coordination → no-op, Retrospective → conversational prompt.
* **3.15 ACP-Compliant Task Payload Consumption** — parse the JSON `task.input` contract from EPIC-02 §2.13 into typed `acp.SessionPromptParams`; resolve `context_files` under `project.WorkingDir`; reject path escapes.
* **3.16 YOLO Auto-Approval Bypass** — short-circuit Confirmation/Retrospective tasks whose `assignee_id` is `sys-auto-approve`; record the bypass in `task.output` for audit.

### 5.2 EPIC-04 Wiring (Stories 3.17–3.18)
* **3.17 CUI Adapter Activation in Daemon Mode** — `--mode=daemon` constructs and registers `CUIAdapter` against the mediator (currently only TUI is registered, regardless of mode); reserves a `NotificationPusher` seam for Telegram/Slack transports.
* **3.18 Mediator First-In-Wins Broadcast** — `PromptDecision` fans out to all registered adapters concurrently; first response wins; losers receive `CancelPrompt`. Adds the `freeform` prompt kind required by Retrospective tasks.

### 5.3 Cross-Interface Testing Affordance (Stories 3.19–3.22)
Stories 3.12–3.18 wire EPIC-02 outputs through the ACP layer, but neither EPIC-02 (project workflows) nor EPIC-04 §4.5 (Telegram/Slack transport) needs to ship for us to validate the plumbing end-to-end. The next four stories let a developer ask a non-project question from the TUI and reply from the CUI — exercising worker, mediator, fanout, inbound channels, and the swappable Provider abstraction under realistic load.

* **3.19 System Assistant Conversational Pipeline** — wires a `SystemAssistant` into the dispatcher so freeform (non-slash) text routes to a session-scoped agent that handles non-project Q&A. Replies broadcast to every registered adapter via `EventChatChunk` and persist as Coordination tasks. The backing Provider is resolved from Story 3.22's registry — never hardcoded.
* **3.20 Hybrid TUI + CUI Adapter Mode** — adds `--mode=hybrid` so both adapters register against the same mediator. TUI keeps stdin ownership; the CUI receives broadcasts and feeds replies back via Story 3.21.
* **3.21 Inbound Adapter Input Multiplexer** — generalizes input ingestion: TUI stdin, CUI stub, and (eventually) Telegram all feed an `InputMultiplexer` that the dispatcher reads from. Includes a stub inbound backend so this story is testable without standing up a Telegram bot. EPIC-04 §4.5 plugs the production transport into the same interface.
* **3.22 Assistant Provider Configuration & Registry** — `config.yaml` declares available ACP Providers (OpenCode, Codex, Claude Code, …) and binds logical Assistants (System, Brainstormer, Clarifier, Archivist) to a Provider by name. Built-in catalog ships sensible defaults; OpenCode is the default. Switching the System Assistant Provider becomes a config edit, not a code change.
* **3.23 `assistants:` Config Refactor & Per-Provider Model Override** — drops the unused legacy `agents:` YAML section; makes `assistants:` the only path for declaring per-Assistant Provider/model bindings; teaches `Provider` how to translate a binding's `model` into spawn args via `ModelFlag`. End result: `assistants.system.model: github-copilot/claude-sonnet-4.6` causes the worker to spawn as `opencode acp --model github-copilot/claude-sonnet-4.6` — no code change.
